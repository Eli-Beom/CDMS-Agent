DVS 문서업무 대체를 위한 열 구성 전환 보고서

작성일: 2026-05-28

문제 정의

현재 목표는 기존 DVS Validation 문서를 더 빠르게 작성하거나 검증하는 것이 아니라, 장기적으로 DVS Validation 문서 업무 자체를 제거하는 것이다. 따라서 기존 문서의 열 구조를 그대로 유지하는 방식은 자동화 목표에 한계가 있다.

기존 문서는 사람이 Specification, Test Script, Expected Result를 작성하고, 이후 검증 담당자가 Result와 Comment를 남기는 문서 중심 프로세스에 맞춰져 있다. 반면 CDMS-Agent가 지향하는 구조는 CRF 코드, trigger rule, browser inspect, query discovery loop 결과를 이용해 검증 규칙 후보와 실제 관찰 결과를 자동으로 축적하고 해석하는 방식이다.

수행 배경

기존 접근 첨부된 DVS Validation List를 기준 문서로 보고, generated test script 또는 discovery result가 문서의 Specification/Test Script/Expected Result와 맞는지 확인하는 방식.
변경된 접근 기존 DVS 문서는 정답지가 아니라, 제거해야 할 문서 업무의 예시로 본다. CDMS-Agent는 기존 문서 열을 그대로 복제하지 않고, 자동화 시스템이 직접 판단하고 사람이 승인할 수 있는 새로운 열 구성을 만든다.
핵심 판단 Evidence Source, Discovery Result, Review Status, Comment 같은 열을 기존 문서에 단순 추가하는 방식은 기존 수동 검증 열과 목적이 겹칠 수 있다. 따라서 기존 문서 열을 보존하는 방향이 아니라, 역할 중심의 새 열로 변환하는 방향이 타당하다.

열 구성 전환 원칙

기존 DVS Validation 문서의 열을 그대로 따르지 않고, “검증 규칙 초안 생성 → 브라우저 실행 관찰 → 시스템 해석 → 사람 승인” 흐름에 맞는 새 열 구조로 전환한다.
문서 작성 중심 사람이 Specification, Test Script, Expected Result를 작성하고 수동 결과를 기록하는 구조.
자동화 중심 시스템이 CRF trigger와 browser inspect를 분석하고, discovery loop 실행 결과를 바탕으로 rule draft와 observation을 생성하는 구조.
검증 관점 pass/fail 확정이 아니라, 어떤 규칙이 어떤 입력과 어떤 관찰 결과로 해석되었는지를 추적한다.

제안 열 구조

rule_id자동 생성된 검증 규칙 식별자. 기존 DVS ID의 역할을 자동화 시스템 기준으로 대체한다.
page_idCRF 코드 또는 현재 브라우저 context에서 확인한 page 식별자.
page_label브라우저 inspect에서 확인한 현재 페이지 표시명.
visit_idvisit context. 현재 자동화 범위에서는 현재 방문 또는 CRFRunner가 해석한 방문 정보를 사용한다.
item_idCRF item 식별자. 가능한 경우 trigger/parser 결과와 연결한다.
item_label화면 rowLabel 또는 CRF item label.
rule_typemissing_query, range_query, option_query, visibility, availability, disability 등 규칙 유형.
rule_source규칙 후보가 생성된 근거. 예: crf_trigger, browser_query_observed, crf_and_browser.
specification_draft기존 Specification에 해당하지만, 사람이 작성한 문장이 아니라 시스템이 생성한 규칙 설명 초안.
test_input_steps실제 browser-assisted loop에서 실행한 입력 단계. 기존 Test Script를 실행 가능한 형태로 대체한다.
expected_observation시스템이 실행 전 예상한 관찰 결과. 예: query_expected, no_query_expected, manual_review.
observed_query_count저장 후 inspect에서 확인된 query 개수.
observed_query_messages저장 후 화면에서 실제 관찰된 query 문구 목록.
actual_observation실제 관찰 결과 요약. 예: query_observed, no_query_observed, skipped, blocked.
interpretation규칙 후보, 입력값, 실제 query 문구를 종합한 시스템 해석. 사람이 검토할 핵심 설명이다.
confidence자동 해석 신뢰도. 예: high, medium, low.
review_decision사람의 최종 승인 판단. 예: accepted, rejected, needs_edit, defer.
review_comment승인자 또는 검토자가 남기는 수정 의견.

기존 문서 열과의 관계

DVS ID rule_id로 전환한다. 사람이 부여하는 문서 ID보다 자동화 시스템에서 재생성 가능한 규칙 ID가 필요하다.
Specification specification_draft로 전환한다. CRF trigger와 browser evidence를 바탕으로 시스템이 작성하는 규칙 설명이다.
Test Script test_input_steps로 전환한다. 문장형 테스트 절차가 아니라 실제 실행된 agent 입력 명령을 남긴다.
Expected Result expected_observation로 전환한다. 문서상 기대 결과가 아니라 시스템이 실행 전 예상한 관찰 결과를 표현한다.
Result / Comment actual_observation, interpretation, review_decision, review_comment로 역할을 분리한다. 자동 관찰 결과와 사람 승인 결과를 같은 열에 섞지 않는다.

현재 결정 사항

  • 기존 DVS 문서를 기준 문서로 검증하는 방향은 보조적인 관점으로만 둔다.
  • 최종 목표는 DVS Validation 문서 업무를 제거하는 것이다.
  • 이를 위해 기존 문서 열을 그대로 복제하지 않고 역할 중심 열로 변환한다.
  • browser-assisted query discovery loop의 결과는 새 산출물의 observed_query_count, observed_query_messages, actual_observation, interpretation에 반영한다.
  • 사람의 업무는 문서 작성이 아니라 review_decision과 review_comment 중심의 승인/수정으로 축소한다.

다음 작업

1. 산출물 모델 정의 RuleDiscoveryRecord 또는 유사한 데이터 모델을 Python package에 정의한다.
2. notebook 출력 변경 DISCOVERY_RESULT를 기존 query count 요약에서 새 열 구조 기반의 rule discovery table로 확장한다.
3. 해석 로직 추가 CRF trigger, 입력 step, observed query message를 종합해 interpretation과 confidence를 생성한다.
4. export 추가 현재 페이지 기준 rule discovery table을 CSV 또는 XLSX로 내보낼 수 있게 한다.
5. 승인 흐름 분리 자동 생성 결과와 사람 승인 결과를 분리해, 문서 작성이 아닌 검토/승인 중심으로 업무를 축소한다.

요약

이번 결정은 DVS 문서를 더 잘 작성하기 위한 개선이 아니라, DVS 문서 업무를 제거하기 위한 문제 정의와 산출물 구조 전환이다. 기존 문서의 열을 참고 대상으로만 보고, CDMS-Agent가 생성·실행·관찰·해석·승인 흐름을 직접 관리할 수 있는 새 열 구성을 채택하기로 했다.
