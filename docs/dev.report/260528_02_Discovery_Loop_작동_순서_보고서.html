Discovery Loop 작동 순서 보고서

작성일: 2026-05-28

대상: browser-assisted-query-discovery / CDMS-Agent_test_v0.4.6.ipynb

목적

Discovery loop의 목적은 CRF rule만으로 query 발생 여부를 단정하는 것이 아니라, 현재 브라우저 페이지에서 입력 후보를 실제로 실행하고 저장한 뒤, 저장 후 화면 inspect 결과를 통해 query 발생 여부를 관찰하는 것이다.

이 결과는 pass/fail 판정이 아니라 DVS 문서업무 대체를 위한 Rule Discovery 산출물의 관찰 데이터로 기록된다.

전체 흐름

candidate 생성 → 사용자 확인 → 입력 step 실행 → Save 전 사용자 확인 → Save 실행 → 대기 → inspect 실행 → query row 수집 → Rule Discovery 결과 행 생성 → 표 출력 및 CSV export

상세 작동 순서

1. 현재 페이지 context 확인 notebook 생성 시 agent.inspect()로 현재 브라우저 페이지의 page_id, page_label, pathname, visible_rows, structured_rows를 가져온다.
2. candidate 생성 CRFRunner.query_cases()에서 현재 페이지에 해당하는 query/no-query 후보를 가져오고, browser inspect에서 확인된 row와 매칭 가능한 입력 step만 candidate로 구성한다. 매칭 실패 시에는 현재 화면의 기본 입력값을 사용하는 fallback candidate를 만든다.
3. Rule Discovery 메타데이터 부여 각 candidate에는 DVS ID, item_id, item_label, rule_type, rule_source, Specification, Expected Result, steps가 포함된다.
4. candidate 실행 전 정지 loop는 각 candidate 실행 전에 input(...)으로 멈춘다. 사용자는 현재 화면 상태를 확인한 뒤 Enter를 눌러 해당 후보 실행을 시작한다.
5. 입력 step 실행 candidate의 steps를 순서대로 실행한다. 예: agent.set_date(...), agent.select_radio(...), agent.set_text(...). 각 step 결과는 내부적으로 성공/실패/skip 상태를 수집한다.
6. Save 전 정지 모든 입력 step 실행 후 input('Press Enter to Save and observe query...')에서 다시 멈춘다. 사용자는 저장 전 화면 피드백과 입력값을 육안으로 확인할 수 있다.
7. Save 실행 Enter 입력 후 agent.click_save(page_id=DISCOVERY_PAGE_ID, visit_id=DISCOVERY_VISIT_ID)를 실행한다. Save raw result는 출력하지 않는다. 저장 중 오류가 발생하면 save_error로만 기록한다.
8. 저장 후 대기 POST_SAVE_DELAY_SECONDS만큼 대기한다. 이는 저장 직후 query row가 화면에 반영되는 시간을 확보하기 위한 것이다.
9. 저장 후 inspect 저장 후 agent.inspect()를 다시 실행한다. 이 시점의 inspect 결과가 실제 query 발생 여부 판단의 기준이 된다.
10. query row 수집 query는 다음 source에서 수집한다.
snapshot.query_rows
snapshot.visible_rows 중 Query [ 포함 row
snapshot.invalid_row_labels
11. Result 산출 observed_query_count가 1 이상이면 Result = query_observed, 0이면 Result = no_query_observed로 기록한다. Save/click 문제가 있으면 blocked, runner skip이 있으면 skipped로 기록한다.
12. Rule Discovery 행 생성 각 candidate 실행 결과는 DISCOVERY_RESULTS에 누적되고, 최종적으로 rule_discovery_result DataFrame으로 변환된다.
13. 결과 표 분류 loop 종료 후 다음 표를 출력한다.
전체 rule_discovery_result
query_observed
no_query_observed
manual_review_or_unknown
review_queue
14. CSV export 결과가 비어 있지 않으면 현재 페이지 기준 CSV를 저장한다. 예: CDMS-Agent_rule_discovery_DM_v0.1.csv

결과 열 구조

DVS ID자동 생성된 검증 규칙 후보 식별자.
page_id / page_label / visit_id현재 실행 context.
item_id / item_label규칙 후보와 연결된 CRF item 또는 화면 row.
rule_type규칙 유형. 예: query, range_query, option_query, unknown.
rule_source규칙 후보 생성 근거. 예: CRF, Browser, CRF+Browser, Unknown.
SpecificationCRF trigger/parser 기반 조건식 또는 규칙 설명 초안.
Test Script실제 loop에서 실행한 agent 입력 step.
Expected Result실행 전 시스템 예상값. 예: query_expected, no_query_expected, manual_review.
observed_query_count저장 후 inspect에서 관찰된 query 개수.
observed_query_messages저장 후 inspect에서 관찰된 query 문구.
Result실제 관찰 결과. 예: query_observed, no_query_observed, skipped, blocked.
review_decision사람이 나중에 승인/보류/수정 여부를 채우는 열. 자동 생성 시 비워둔다.

중요 기준

이 loop는 query 발생 여부를 자동으로 검증하는 assert가 아니다. 실제 Save 후 inspect로 관찰한 결과를 기록하는 discovery/observation 흐름이다. 따라서 Result는 pass/fail이 아니라 현재 화면에서 관찰된 상태를 의미한다.

제외한 항목

  • Save result raw 출력은 하지 않는다.
  • check_result, CASE_CHECK, has_query 기반 검증은 사용하지 않는다.
  • Comment, confidence, interpretation, review_comment는 생성하지 않는다.

요약

v0.4.6 discovery loop는 candidate별 입력을 실제 브라우저에서 실행하고, 저장 후 inspect를 통해 query row를 수집한다. 수집 결과는 DVS 문서업무 대체용 Rule Discovery 산출물로 정리되며, 사람의 역할은 문서 작성이 아니라 결과 확인과 review_decision 입력으로 축소된다.
