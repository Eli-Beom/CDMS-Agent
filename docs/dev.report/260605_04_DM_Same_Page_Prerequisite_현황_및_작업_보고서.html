DM Same-page Prerequisite 현황 및 작업 보고서

작성일: 2026-06-05

대상: DM / 인구학적 조사

1. 현재 판단

직전 v0.6.9에서는 setup_steps를 main query loop 안에서 자동 실행하도록 넣었고, 이 때문에 정확도가 더 떨어진 것으로 판단한다. prerequisite 입력은 query 발생 조건 자체가 아니라 target row를 활성화하기 위한 선행 상태 설정이므로, query 검증 루프와 섞으면 입력 상태와 저장 타이밍이 불안정해진다.

따라서 v0.6.10부터는 prerequisite을 candidate metadata와 노트북의 별도 표시 표로만 제공하고, 기본 query loop에서는 자동 실행하지 않도록 변경했다.

2. 확인된 원인

문제영향조치
같은 페이지 내 선행조건 필요 BMLOC, BSLOC 같은 target row가 비활성 상태이면 입력 및 저장이 실패할 수 있음 candidate에 setup_steps, prerequisite_item_ids metadata 추가
main query loop에서 setup 자동 실행 query 검증 전 상태가 바뀌고, row 선택 실패 또는 저장 실패가 더 잘 발생할 수 있음 v0.6.10에서 기본값 EXECUTE_SETUP_STEPS = False로 변경
반복 label row BMLOC와 BSLOC가 화면상 모두 부위로 보여 row label만으로는 정확히 구분하기 어려움 후속 작업으로 section/item id 기반 row resolver 필요
query 미감지 실제 화면에는 query가 떠도 결과가 observed_query_count = 0이 될 수 있음 extension wait-query 방식은 추가했으나, 개별 candidate 입력 성공 여부와 저장 이후 DOM 변화를 더 추적해야 함

4. 적용된 변경

파일변경 내용
candidate_prerequisites.py 공통 helper enrich_same_page_prerequisites() 추가. DM 전용이 아니라 모든 page candidate 생성 경로에서 사용 가능.
rule_discovery.py browser/static candidate 생성 시 same-page prerequisite metadata를 enrich하도록 연결.
generate_screening_rule_discovery.py screening page candidate 생성 시 공통 prerequisite helper 사용. notebook 버전 v0.6.10 생성.
rule_discovery_runtime.py execute_setup_steps 옵션 추가. 기본값은 False이며, setup은 main query loop에서 자동 실행하지 않음.

5. v0.6.10 노트북 동작

EXECUTE_SETUP_STEPS = False

rule_discovery_result = run_rule_discovery_loop(
    agent,
    DISCOVERY_CANDIDATES,
    output_csv_path=OUTPUT_CSV_PATH,
    execute_setup_steps=EXECUTE_SETUP_STEPS,
)

노트북은 loop 실행 전에 prerequisite 후보 표를 표시한다. 사용자는 이 표에서 선행조건이 필요한 candidate를 먼저 확인하고, 필요하면 별도 셀 또는 수동 상태 설정 후 query loop를 실행하는 구조로 운용한다.

7. 남은 리스크

이번 수정은 v0.6.9에서 정확도를 떨어뜨린 main loop 자동 setup 실행을 되돌리고 분리한 것이다. 그러나 D_DM_5, D_DM_7_1, D_DM_14_1의 실제 query 감지 실패는 아직 완전히 해결된 것이 아니다.
  • 부위처럼 label이 중복되는 row는 section/item id 기반 selector가 필요하다.
  • query candidate별 입력 step이 실제로 화면에 반영됐는지 step outcome을 기록해야 한다.
  • 저장 후 query DOM이 나타나는 타이밍을 extension wait-query와 snapshot 양쪽에서 비교해야 한다.
  • availability 검증은 query 검증 loop와 분리된 별도 loop로 운영해야 한다.
