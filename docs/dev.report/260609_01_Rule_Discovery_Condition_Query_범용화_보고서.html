CDMS-Agent Rule Discovery Condition Query 범용화 보고서

목적

Rule Discovery candidate를 특정 DVS ID나 특정 과제에 맞춰 하드코딩하지 않고, CRF TypeScript의 trigger.ts에 정의된 conditional 구조를 해석해 생성하도록 정리했다. 이번 버전의 notebook은 v0.7.0이다.

범용 해석 원칙

항목설명
원천CRF TypeScript trigger의 conditional, issue, page field/option map을 사용한다.
단순 비교A = 1, A != 2 같은 leaf expression을 condition_items로 추출한다.
AND 조건같은 candidate 안에 모든 condition step을 포함한다.
OR branch(A OR B) AND C를 A AND C, B AND C 형태의 branch candidate로 분리한다.
UI 값 변환DB value는 page field option map을 통해 UI label로 변환한다.
하드코딩 금지D_DM_7_1 같은 특정 DVS ID 조건문을 코드에 직접 넣지 않는다.

Candidate 필드

  • query_category: condition_query, range_query, missing_query 등 query 발생 원인 분류
  • condition_type: single, multi, cross, derived, unsupported
  • condition_items: trigger conditional에서 추출한 item/operator/value/ui_value 목록
  • condition_steps: 실제 browser automation 실행 step
  • condition_branch_index, condition_branch_count: OR branch가 분리된 경우 branch metadata

D_DM_7_1 검증 예시

D_DM_7_1은 아래 조건을 가진다.

(OPINDCL = 2 OR OPINDCR = 2) AND BMYN = 2

Runtime Loop

기존 run_rule_discovery_loop()를 유지한다. query_observation_mode="per_candidate"에서는 candidate 실행 전후 query snapshot diff를 사용하고, sweep mode에서는 누적 query를 정리한다.

Notebook v0.7.0

Screening page별 notebook은 CDMS-Agent_test_<PAGE_ID>_v0.7.0.ipynb 이름으로 생성된다. Cell은 query category별로 분리되어 있으며, Condition Query - Multi 셀은 branch candidate를 loop로 실행할 수 있다.

남은 제한

  • 복잡한 nested OR/AND가 과도하게 커지는 경우 branch explosion 제한이 필요하다.
  • appendable table 내부 row/column control은 별도 개발 대상이다.
  • cross-page condition은 현재 page 단독 자동 실행 대상에서 분리해야 한다.
  • custom function, item-to-item 계산 비교는 추가 parser가 필요하다.
