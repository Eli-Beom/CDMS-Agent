CDMS-Agent Query Category Validation Loop 변경 보고서

배경

기존 Rule Discovery loop는 모든 query candidate를 같은 방식으로 실행하고, 저장 후 화면에 남아 있는 query 중 candidate label과 매칭되는 항목만 결과로 사용했다. 이 방식은 화면에 남아 있는 이전 query, 실제 query label과 candidate target label 불일치, missing/range/calculation/조건 조합 query 혼재 때문에 PASS/FAIL 해석이 불안정했다.

변경 요약

영역변경 내용
Candidate taxonomyquery_category, condition_type를 query candidate에 추가했다.
Candidate steps신규 실행 로직은 condition_steps를 우선 사용하고, 기존 steps는 backward compatibility로 유지한다.
Runtime loop기존 run_rule_discovery_loop()를 유지하고 query_observation_mode 옵션을 추가했다.
per_candidate modecandidate 실행 전/후 query snapshot을 비교해 새로 발생한 query만 관찰 대상으로 사용한다.
sweep modeloop 시작/종료 snapshot diff로 누적 query를 정리하고, 결과는 validation PASS/FAIL과 분리한다.
NotebookDM notebook을 CDMS-Agent_test_DM_v0.6.16.ipynb로 생성하고 category별 셀을 물리적으로 분리했다.

Query Category 기준

  • missing_query: 입력 누락 또는 required 기반 query
  • condition_query: 특정 condition item 값이 충족될 때 발생하는 query
  • range_query: 허용 범위 또는 방문 window 이탈 query
  • calculation_query: 자동 계산/derived item 검증 query
  • consistency_query: 여러 item 간 의미 불일치 query
  • unsupported_query: 자동화 불가 또는 해석 제한 query

Condition Type 기준

  • none: condition item 없음
  • single: 같은 page condition item 1개
  • multi: 같은 page condition item 2개 이상
  • cross: 다른 page/visit/section 참조 필요
  • derived: 계산/자동 산출값 참조
  • unsupported: 자동 실행 불가

남은 리스크

  • OR branch는 현재 simple comparison metadata를 보존하지만 branch별 candidate 분리는 아직 제한적이다.
  • appendable table 내부 row/column control은 별도 개발 대상이다.
  • query message label과 DVS issue/condition label이 다르게 표시되는 경우 sweep 결과를 기준으로 추가 mapping rule이 필요할 수 있다.
  • no_query_expected 검증은 page cleanup/reset 전략이 없으면 이전 query 잔존 영향을 받을 수 있어 per_candidate diff를 우선 사용해야 한다.
