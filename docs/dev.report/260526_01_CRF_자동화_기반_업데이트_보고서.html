CRF Automation Update Report

Project: CDMS-Agent Date: 2026-05-26 (_260526) Scope: CRF test script generation foundation

어제 업데이트의 핵심은 과제별 CRF가 제각각인 상황에서, 무리하게 모든 rule을 자동화하기보다 CRF 소스 진단, query 후보 분류, notebook 생성, 문서화의 기본 골격을 갖추는 것이었습니다.

개발 의도

  • 과제마다 다른 label, option, trigger, precondition 때문에 테스트 스크립트를 하드코딩하지 않는 구조가 필요했습니다.
  • 최근 문제였던 중복 trigger id를 빠르게 잡기 위해 doctor 진단 기능을 우선 강화했습니다.
  • DVS rule 전체 자동 변환은 위험하므로, 우선 자동화 가능한 query trigger와 수동 검토 대상을 나누는 방향으로 정리했습니다.
  • notebook은 한 번에 끝나는 자동화가 아니라, 사람이 셀 단위로 실행하며 브라우저 피드백을 확인하는 작업 공간으로 설계했습니다.

처리 흐름

1. CRF Sourcepage.ts, trigger.ts 읽기
2. Doctor중복 id, 누락 참조, query 합계 진단
3. Runner자동화 가능한 query case 생성
4. Notebook셀 단위 검증 스크립트 출력

예시 과제 진단 관점

예시 과제 20260325_PRACTICE_GSB는 query trigger 수가 많고 dynamic context도 섞여 있어, 전체를 자동화 대상으로 보는 대신 “자동화 가능한 query”와 “manual review”를 나누는 기준이 필요했습니다.

  • query triggers: 전체 query 후보 규모 파악
  • query auto: notebook으로 생성할 수 있는 후보
  • query manual review: dynamic/repeated context 때문에 사람이 봐야 하는 후보
  • availability items: 화면 활성/비활성 조건이 있는 항목 규모 파악

판단한 한계

  • DVS rule을 코드로 1:1 변환해 브라우저에서 완전 자동 검증하는 것은 현재 CRF 복잡도상 위험합니다.
  • 브라우저에서 query 발생 여부는 저장 후 확인되므로, 너무 빠른 자동 저장은 피드백 확인을 어렵게 합니다.
  • 정적 CRF 분석만으로는 현재 subject/visit/page의 실제 UI 상태를 알 수 없습니다.
