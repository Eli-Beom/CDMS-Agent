1. 결론

현재 CDMS-Agent는 cross-page/query 확장을 완전히 새로 만들 필요가 없다. 이미 go_to_page, PRECOND/MAIN step, AGE generator, field.calculate 추출, date diff variant 일부가 존재한다. 따라서 첫 단계는 신규 dependency graph 대형 구현이 아니라, 기존 runner/simulator 결과를 audit하고 taxonomy metadata를 보강하는 것이다.

2. 현재 코드에 이미 있는 기능

기능현재 위치상태의미
페이지 이동 cdm_agent_client/client.py / CDMSAgent.go_to_page() 이미 있음 agent.go_to_page("DM"), agent.go_to_page("V2/DM") 형태 지원.
cross-page precondition step crf/runner.py / build_query_cases_for_trigger() 일부 있음 PRECOND, MAIN 입력을 page별 step으로 만들 수 있음.

date diff variant crf/simulator.py / valAs == "DAYS" 일부 있음 BUDAT - SVDAT, BQPRDAT - SVDAT 같은 window rule의 경계값 생성 기반이 있음.
field.calculate 추출 crf/extractor.py, crf/models.py 이미 있음 AGE, Breast Q 총합, RASCH 변환점수 등 page field 계산식을 spec으로 추출함.

4. 우선 구현 원칙

  1. 신규 대형 모듈보다 기존 생성 결과 audit을 먼저 한다.
  2. candidate JSON schema는 점진적으로 확장한다.
  3. query, availability, calculation loop는 분리하되 기존 run_rule_discovery_loop() 호환성은 유지한다.
  4. CSV export는 만들지 않는다.
  5. Result는 PASS/FAIL 중심으로 유지한다.
  6. 다른 page/visit 이동이 필요한 경우 자동 이동 가능 여부와 수동 pause 필요 여부를 구분한다.

5. Phase별 구현 계획

Phase 0. 현 상태 Audit

목표: 실제 CRF 코드에서 CDMS-Agent가 이미 무엇을 생성하는지 확인한다.

  • runner.query_cases()가 D_DM_1_2, D_PY_3, D_BU_2, D_BQ_2에 대해 만드는 step을 출력한다.
  • 실패를 세 가지로 분리한다: 생성 실패, candidate 필터링 실패, runtime 실행/관찰 실패.
  • field.calculate로 추출되는 AGE, Breast Q 총합, RASCH 변환점수를 audit table로 출력한다.
audit targets:
- D_DM_1: AGE calculation
- D_DM_1_2: AGE<22 query
- D_PY_3: HEIGHT range query
- D_BU_2: BUDAT-SVDAT date window query
- D_BQ_13: Breast Q sum calculation
- D_BQ_14: RASCH auto-link
- D_BQ_2: BQPRDAT-SVDAT date window query

Phase 1. Taxonomy Metadata 추가

목표: 현재 candidate를 유지하면서 분류 정보만 명확히 추가한다.

new metadata:
- validation_category
- query_subtype
- calculation_subtype
- automation_scope
- expected_query_message_label
- depends_on_item_id
- expected_calculated_value
- limitation_reason

이 단계에서는 dependency graph를 완성하지 않는다. 기존 candidate에 metadata enrichment를 먼저 적용한다.

Phase 2. Query / Availability Loop 분리

목표: 결과 해석을 단순화한다.

  • run_query_discovery_loop(): query/range/date/option consistency 전용.
  • run_availability_discovery_loop(): availability 전용.
  • 기존 run_rule_discovery_loop()는 wrapper로 유지.
  • 출력 표에서 내부 target 컬럼은 숨긴다.

Phase 3. Calculation Audit 및 Runtime 기반

목표: 자동계산/자동연동 검증 준비를 한다.

  • collect_calculation_items() 추가.
  • _extract_row_value(snapshot, row_label) 구현.
  • AGE, Breast Q 총합, RASCH를 calculation candidate 또는 audit record로 만든다.
  • 초기에는 calculation을 실행하지 않고 audit/report로만 출력해도 된다.

Phase 4. Calculation-derived Query 지원

목표: AGE처럼 계산값 기반 query를 안정적으로 검증한다.

D_DM_1:
  RFICDAT + BRTHDAT -> AGE

D_DM_1_2:
  AGE<22 -> Query [나이]
  • query candidate 안에 calculation dependency metadata를 둔다.
  • runtime에서 expected calculated value를 중간 확인한 뒤 query 관찰로 넘어간다.
  • 계산값 mismatch와 query mismatch를 구분해 기록한다.

Phase 5. Cross-page / Cross-visit Query 지원

목표: setup page 입력 후 target page query를 관찰한다.

workflow:
1. setup_steps_before_target 실행
2. 필요한 page/visit로 go_to_page 또는 pause
3. source 값 입력
4. 필요 시 save
5. target page 이동
6. target 값 입력
7. save
8. Query [target label] 관찰
9. PASS/FAIL 판정

이 단계에서 필요하면 dependency_graph.py와 cross_page_planner.py를 만든다.

Phase 6. Appendable Table / Complex COUNT 지원

목표: AE/MH 등 반복 테이블과 COUNT/ROW/SIBLING 조건을 다룬다.

  • append row 생성 또는 row index 선택 기능 필요.
  • SIBLING, ROW_INDEX, COUNT(...) 조건 해석 필요.
  • 초기에는 manual review로 두고, 별도 개발 과제로 분리한다.

11. 최종 권장 사항

이번 확장은 dependency graph를 바로 새로 만드는 프로젝트가 아니라, 기존 CDMS-Agent의 runner/simulator가 이미 생성하는 정보를 보존하고 metadata화하는 작업부터 시작해야 한다. 그래야 현재 동작하는 DM/PE/BU 시연 흐름을 깨지 않고 cross-page, calculation, auto-link로 확장할 수 있다.
