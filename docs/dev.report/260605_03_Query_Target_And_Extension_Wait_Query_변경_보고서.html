1. 변경 배경

DM 페이지 Rule Discovery 실행 결과에서 query가 실제로 발생했을 가능성이 있는데도 observed_query_count = 0으로 기록되는 문제가 확인되었다. 특히 D_DM_5는 CRF 스펙상 좌측/우측 유방 수술 목적을 모두 해당사항 없음으로 입력하면 query가 발생해야 하지만, 기존 candidate는 첫 입력 row인 [좌측 유방] 수술 목적을 query target처럼 사용했다.

실제 CRF trigger의 issue.itemId는 OPINDCR이며, query message label은 [우측 유방] 수술 목적을 기준으로 뜰 가능성이 높다. 따라서 문제는 개별 candidate 수정보다 candidate 생성 로직의 target 결정 기준에 있었다.

2. 기존 문제

영역기존 동작문제
Candidate 생성 첫 번째 입력 step의 row label을 item_label로 사용 query target이 아닌 입력 row를 target으로 오인
Query message 매칭 item_label, item_id, step label fallback 기준 매칭 실제 issue target label과 다르면 query를 못 셈
Query 감지 시점 Save 후 sleep 후 inspect 1회 query 표시 타이밍에 따라 감지 누락 가능

4. Runtime Query Matching 변경

runtime의 query matching은 이제 expected_query_message_labels와 expected_query_message_label을 우선 사용한다. 해당 값이 없을 때만 기존 item_label, item_id, step label fallback을 사용한다.

이 변경으로 D_DM_5처럼 입력 row와 query target row가 다른 경우에도 실제 query message label 기준으로 query count를 계산할 수 있다.

5. Extension-side Wait Query 추가

기존 방식은 Save 후 Python에서 sleep 후 inspect를 1회 수행했다. 새 방식은 Chrome extension의 page context에서 MutationObserver로 query message DOM 변화를 기다린다. 

notebook runtime
  -> agent.click_save()
  -> agent.wait_query(labels=[expected_query_message_label])
  -> daemon /api/cdm-agent/wait-query
  -> extension command: wait_for_query
  -> browser-runner-core.js MutationObserver
  -> Query message 발견 즉시 Python으로 반환
구성요소변경 내용
browser-runner-core.js waitForQueryMessages(labels, timeoutMs) 추가, runner version 0.0.14-query-event-push
service-worker.js wait_for_query command 처리 추가
daemon/src/index.ts POST /api/cdm-agent/wait-query endpoint 추가
client.py CDMSAgent.wait_query() 추가
rule_discovery_runtime.py Save 후 extension-side wait query를 우선 사용하고 snapshot query와 병합

8. 실제 사용 시 필요한 조치

변경된 extension runner와 service worker를 사용하려면 daemon 재시작, Chrome extension reload, CDMS page refresh가 필요하다. 이후 CDMS-Agent_test_DM_v0.6.8.ipynb에서 D_DM_5를 먼저 실행해 실제 query 감지 여부를 확인한다.

9. 남은 리스크

D_DM_14_1처럼 서로 다른 item이 화면에서 같은 label(부위)로 표시되는 경우, query message도 단순히 Query [부위]로 뜰 수 있다. 이 경우 label만으로는 row 구분이 불충분하므로 향후 section/item id 기반 row matching 보강이 필요하다.
