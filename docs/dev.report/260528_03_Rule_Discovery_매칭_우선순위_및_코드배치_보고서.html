Rule Discovery 매칭 우선순위 및 코드배치 보고서

작성일: 2026-05-28

대상: CDMS-Agent browser-assisted query discovery / Rule Discovery 산출물

배경

현재 Rule Discovery 기능은 브라우저 inspect 결과, CRF page/item 정의, trigger 조건, query 관찰 결과를 함께 사용한다. 그러나 최근 개발은 notebook 생성 결과를 빠르게 개선하는 데 집중되어, 판정 로직이 src/cdm_agent_client/crf/notebook.py 내부에 많이 섞여 있다.

이 상태가 지속되면 page 매칭, trigger 후보 수집, rowLabel 정규화, boundary value 생성, discovery result 출력이 한 파일에 얽히게 되어 유지보수와 과제별 확장이 어려워진다.

매칭 우선순위

1. 현재 browser pathname의 page_id 현재 작업 대상 페이지를 확정하는 1차 기준이다. 예: /.../VS/1이면 현재 page는 VS로 본다.
2. CRF 내부 crfPageId/pageId CRF 코드 내부의 실제 page 참조를 확인한다. 파일명이 아니라 crfPageId: "VS", pageId: "VS" 같은 구조화된 값을 우선한다.
3. itemId page 안의 실제 field/item을 연결한다. 예: VS2SBP, VS2DBP, VS2TEMP.
4. item label / rowLabel CRF item label과 브라우저 inspect의 rowLabel을 연결한다. bracket, visit prefix, 공백, 기호를 제거한 normalized match도 사용한다. 예: [수술 전] 수축기 혈압과 수축기 혈압.
5. trigger issue.itemId query가 실제로 걸리는 target item을 확인한다. issue.itemId가 현재 page item을 가리키면 관련 trigger 후보로 본다.
6. trigger conditional left/right item ref query 조건에 참여하는 item을 확인한다. conditional.left, conditional.right 안의 crfPageId, itemId를 재귀적으로 탐색한다.
7. 파일명 page_XXX / trigger_XXX 파일명은 보조 힌트로만 사용한다. 최종 판단은 코드 내부의 pageId, crfPageId, itemId, issue, conditional 기준으로 한다.

판정 흐름

1. 현재 페이지 확정 agent.inspect().pathname에서 현재 page_id를 추출한다.
2. page item 후보 수집 CRF page 정의에서 현재 page의 itemId, label, control type을 field map으로 만든다.
3. trigger 후보 수집 현재 page와 직접 연결된 trigger뿐 아니라, D_<page_id>_* id prefix, issue, conditional, bindTo/value/target 참조까지 확장 검색한다.
4. browser row 매칭 browser inspect의 structured_rows와 CRF item label을 매칭한다. exact match가 실패하면 bracket/visit prefix/공백/기호 제거 후 비교한다.
5. candidate 생성 여부 결정 trigger, item, browser row가 모두 연결되면 실행 가능한 candidate로 만든다. trigger는 있으나 row가 없거나 조건이 복잡하면 manual review 후보로 남긴다. browser row는 있으나 trigger가 없으면 browser-only fallback 후보로 둔다.
6. boundary value 생성 단순 비교식이면 query 발생값과 미발생값을 만든다. 예: VS2SBP < 80이면 query 후보 79, 정상 후보 80 이상 값.
7. loop 실행 및 관찰 candidate 입력 후 Save를 실행하고, 저장 후 agent.inspect()로 queryRows/queryCount를 수집한다.
8. Rule Discovery 결과 기록 결과는 pass/fail이 아니라 query_observed, no_query_observed, skipped, blocked 같은 observation으로 기록한다.

현재 코드 위치

Notebook 생성 src/cdm_agent_client/crf/notebook.py
CRF scenario 생성 src/cdm_agent_client/crf/runner.py
trigger condition parse src/cdm_agent_client/crf/parser.py
CRF source 추출 src/cdm_agent_client/crf/extractor.py
브라우저 현재 화면 inspect extension/browser-runner-core.js 및 daemon 배포본

문제점

  • notebook.py가 notebook cell 생성뿐 아니라 row 매칭, fallback value, rule result shaping까지 담당하고 있다.
  • trigger 후보 수집 기준이 page direct match 중심에서 id prefix/issue/conditional 참조까지 확장되어야 하는데, 이 로직이 명시적인 모듈로 분리되어 있지 않다.
  • query message, boundary value, trigger reference 탐색이 여러 계층에 흩어질 가능성이 있다.
  • 과제별 CRF 구조가 다를수록 notebook 생성기 안의 임시 보정이 늘어날 위험이 있다.

코드 분리 제안

Rule Discovery 관련 판정 로직은 notebook.py에서 분리하는 것이 맞다. notebook.py는 최종 notebook cell을 조립하는 얇은 계층으로 유지하고, page/trigger/item/row 매칭과 candidate 생성은 별도 모듈로 옮긴다.

권장 파일 구조

crf/rule_discovery.py Rule Discovery의 상위 orchestration 계층. 현재 page context, CRF spec, browser inspect를 받아 RuleDiscoveryCandidate와 결과 schema를 만든다.
crf/trigger_matcher.py 현재 page 기준 trigger 후보 수집. pageId/crfPageId, D_<page>_* id prefix, issue, conditional, bindTo/value 참조 탐색을 담당한다.
crf/row_matcher.py CRF item label과 browser rowLabel 매칭. bracket 제거, visit prefix 제거, 공백/기호 제거, option 포함 매칭 같은 label normalization을 담당한다.
crf/value_planner.py trigger conditional을 바탕으로 query_expected/no_query_expected 입력값을 생성한다. range boundary, option condition, missing query 후보 생성을 담당한다.
crf/notebook.py 위 모듈에서 생성한 candidate/result schema를 받아 notebook code cell과 markdown cell만 생성한다. 판정 로직은 최소화한다.

권장 데이터 모델

RuleDiscoveryCandidate DVS ID, page_id, item_id, item_label, rule_type, rule_source, Specification, Expected Result, steps를 보관한다.
RuleDiscoveryResult candidate 실행 후 observed_query_count, observed_query_messages, Result, review_decision을 보관한다.
TriggerMatch trigger가 현재 page와 연결된 이유를 내부적으로 보관한다. 예: page_id, id_prefix, issue_ref, conditional_ref. 본문 산출물 열로 노출할 필요는 없다.

정리 방향

  1. 현재 notebook.py 안의 discovery candidate 생성 함수를 rule_discovery.py로 이동한다.
  2. trigger 후보 수집은 trigger_matcher.py로 분리한다.
  3. rowLabel 정규화와 browser row 매칭은 row_matcher.py로 분리한다.
  4. 경계값 생성과 fallback value는 value_planner.py로 분리한다.
  5. notebook.py는 candidate 목록을 받아 notebook loop 코드를 출력하는 역할만 남긴다.

요약

현재 Rule Discovery 기능은 동작 결과를 빠르게 만드는 단계였기 때문에 판정 로직이 notebook 생성 코드에 많이 들어가 있다. 앞으로 과제별 CRF 구조가 달라지는 상황을 감당하려면, page/trigger/item/row/value 판정 로직을 별도 모듈로 분리하는 것이 맞다.

특히 trigger 후보 수집은 현재 page direct match만으로는 부족하므로, D_VS_* 같은 trigger id prefix, issue.itemId, conditional.left/right 참조를 모두 사용하는 명시적인 matcher 계층으로 정리해야 한다.
