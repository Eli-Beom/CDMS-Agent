browser-assisted input harness 목표 정리 보고서

작성일: 2026-05-27

현재 변경된 목표

기존에는 CRF 코드의 DVS/query rule을 해석하여 expected result와 query 발생 여부까지 자동 검증하려는 방향이 포함되어 있었다. 그러나 현재 과제의 CRF 구조와 브라우저 저장 피드백 특성상, DVS rule 자동 검증은 안정적으로 추진하기 어렵다. 따라서 현재 목표는 DVS/query 검증이 아니라 연결된 브라우저의 현재 페이지에서 입력 가능한 row를 최대한 정확히 찾아 notebook 입력 script를 생성하는 것으로 재정의한다.

범위에서 제외한 것

DVS/query 자동 검증현재 단계에서는 보류한다. query 발생 여부는 저장 후 화면 피드백을 사람이 확인한다.
query expected resultnotebook cell에 생성하지 않는다.
has_query, check_result, CASE_CHECK자동 판정 신뢰도가 낮아 제거 상태를 유지한다.
query 합계/분류현재 입력 harness 목적과 직접 관련이 없으므로 생성/요약하지 않는다.

해결 전략

앞으로의 구조는 CRF 코드만으로 test script를 추론하지 않는다. CRF 코드는 page/visit 구조와 기본 context를 잡는 데 사용하고, 실제 입력 가능한 항목 판단은 브라우저 DOM inspect 결과를 우선한다.

Python notebook 생성기 src/cdm_agent_client/crf/notebook.py는 browser-assisted 모드에서 현재 브라우저의 structured_rows를 읽고 입력 step을 생성한다. notebook에는 복잡한 if문이나 사전 row 상태 helper를 넣지 않고, agent.set_date, agent.select_radio, agent.select_option 같은 직접 입력 명령만 둔다.
Extension runner browser-runner-core.js가 실제 DOM 상태를 판단한다. 회사 trigger 용어에 맞춰 row_visibility, row_availability, row_disability를 계산하고, hidden/unavailable/locked row는 실패가 아니라 skipped로 넘긴다.
Radio/option 처리 예, 아니요처럼 여러 row에 반복되는 option 때문에 오클릭이 발생할 수 있다. 따라서 radio 좌표 계산은 document 전체가 아니라 target row 내부에서만 option을 찾아야 한다.
Row 확정 단순 visible text 후보가 아니라 실제 input/control을 기준으로 row를 확정해야 한다. bracket label, table/grid row, section 아래 입력 row를 안정적으로 잡는 것이 다음 핵심 과제다.

현재까지의 노력

query 자동 판정 제거notebook에서 Expected Result, has_query, check_result, CASE_CHECK 흐름을 제거했다.
입력 전용 notebook 생성v0.4.0 notebook을 browser-assisted/current-page 입력 전용으로 생성했다.
runner skip 도입입력 불가 row를 notebook 조건문으로 거르지 않고 extension runner가 skipped 처리하도록 했다.
금지 문자열 확인생성 notebook에 helper/query 자동 판정 문자열이 다시 들어오지 않는지 확인했다.
문제 분리누락 문제를 생성기 누락과 runner 실행 실패로 분리했다. 예: 가임 여부는 생성됐으나 실행 안정성 문제, 좌/우측 유방 수술 목적과 유방암 병력은 row 확정 누락 문제다.

남은 문제

현재 inspect의 structured_rows는 화면에 보이는 모든 입력 row를 안정적으로 확정하지 못한다. 특히 다음 유형이 취약하다.

중복 option row예, 아니요가 여러 row에 반복되어 radio 클릭이 다른 row로 갈 수 있다.
bracket label[좌측 유방] 수술 목적, [우측 유방] 수술 목적처럼 bracket label이 있는 row가 누락될 수 있다.
table/grid 입력부위, 진단일, 병기 확인일, Tumor, Nodes, Metastasis, Stage 같은 table 구조는 rowLabel을 조합해야 한다.
section/header 혼동section title이나 table header가 rowLabel 후보로 섞여 실제 입력 row 확정이 흐려질 수 있다.

다음 개발 방향

  1. structuredRows()를 visible text 기반이 아니라 실제 control 기반으로 재작성한다.
  2. rowLabel은 같은 row의 label/th/td에서 추출하고, option text 단독값은 rowLabel에서 제외한다.
  3. radio 클릭 좌표 계산은 반드시 findRow(rowLabel) 내부에서만 option을 찾도록 수정한다.
  4. CDP radio 클릭 후 target option의 checked 상태를 검증한다.
  5. notebook.py의 _browser_input_cases()는 확정된 structured row를 기반으로 입력 step을 더 많이 생성한다.
  6. v0.4.1 notebook을 생성하여 생년월일, 성별, 가임 여부, 수유 여부, 좌/우측 유방 수술 목적, 유방암 병력 등이 포함되는지 확인한다.

운영 원칙

DVS rule은 나중에 다시 고려한다. 현재는 입력 가능한 row를 찾고 실행하는 harness를 먼저 안정화한다. 저장 후 query 발생 여부는 자동 판정하지 않고, 사람이 브라우저 피드백을 확인한다.
