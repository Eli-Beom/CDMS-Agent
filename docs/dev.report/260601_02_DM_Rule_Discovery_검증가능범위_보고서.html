DM Rule Discovery 검증 가능 범위 보고서

작성일: 2026-06-01 / 대상 과제: 20260325_PRACTICE_GSB / 대상 페이지: DM, 인구학적 조사

1. 개발 배경

현재 DM 프로세스는 EDC development와 EDC validation을 포함해 약 8주가 소요된다. 이 중 EDC validation 단계의 DVS 작성과 DVS validation 업무가 워킹데이 기준 약 20~25일, 즉 약 4주를 차지한다.

CDMS-Agent 개발 목적은 기존 DVS 문서를 그대로 복제하는 것이 아니라, CRF 코드와 실제 EDC 화면을 함께 해석해 DVS ID별 검증 가능성을 자동 산출하고, 반복적인 DVS 작성 및 validation 업무를 줄이거나 제거하는 것이다.

구성요소역할
Python package CRF 코드 파싱, trigger/query/availability 후보 생성, notebook 생성, Rule Discovery runtime 실행을 담당한다.
CRFRunner / Parser page_*.ts, trigger_*.ts를 읽어 page item, item label, trigger, availability spec을 구조화한다.
Rule Discovery JSON 현재 페이지에서 실행할 DVS ID별 후보 목록이다. Query, range_query, availability가 같은 산출물 체계에 들어간다.
Runtime loop JSON candidate를 하나씩 실행한다. Query는 입력 후 Save 및 inspect, availability는 Save 없이 before/after inspect를 수행한다.
Daemon Python client와 browser extension 사이의 API bridge 역할을 한다. Python 명령을 extension runner가 실행할 수 있는 payload로 전달한다.
Browser extension 실제 EDC 화면 DOM을 inspect하고, row 입력, radio 선택, Save 클릭, query row 수집, row_availability 확인을 수행한다.
Generated notebook 사용자가 한 candidate씩 실행하고 화면 피드백을 보면서 Rule Discovery 결과를 축적하는 실행 하네스다.

3. 작동 원리

1Page 인식
browser pathname에서 현재 page_id 확정
2CRF 해석
item, trigger, availability spec 수집
3Row 매칭
EDC rowLabel과 CRF item label 연결
4JSON 생성
DVS ID별 candidate 작성
5Loop 실행
query는 Save, availability는 before/after inspect
6결과 산출
Result는 PASS/FAIL만 기록
단계설명
1. 현재 페이지 인식extension inspect에서 browser pathname과 page label을 읽고 현재 페이지를 DM으로 확정한다.
2. CRF 코드 해석CRF page/trigger 코드에서 item id, label, query trigger, availability spec을 수집한다.
3. 화면 row 매칭현재 화면의 structured_rows와 CRF item label을 매칭한다. exact label, 공백 제거, bracket prefix 제거, 한글 normalize 기준을 사용한다.
4. candidate JSON 생성현재 페이지에서 실행 가능한 query/range/availability 후보를 CDMS-Agent_rule_discovery_candidates_DM_v0.2.json에 저장한다.
5. loop 실행Query candidate는 입력 후 Save 및 inspect를 수행한다. Availability candidate는 Save 없이 control 값을 바꾼 뒤 target row의 availability 변화를 inspect한다.
6. 결과 판정내부 관찰값은 Expected Result와 비교하는 데만 사용하고, 최종 Result에는 PASS 또는 FAIL만 표시한다.

구분CRF 코드 내 DM 기준 총계CDMS-Agent 후보 생성현재 커버리지비고
QUERY 11개 DVS ID 6개 DVS ID 6/11 후보 생성, 그중 일부는 부분 검증 다른 페이지/선행 데이터 참조가 있는 query는 현재 DM 페이지 단독으로 완전 검증하기 어렵다.
ASSIGN 2개 0개 미커버 AGE 등 assign/calculate 결과는 query 관찰에는 사용되지만, assign trigger 자체의 계산 검증은 아직 별도 구현이 필요하다.
AVAILABILITY 5개 item spec 3개 item 3/5 후보 생성 label이 안정적으로 잡힌 PGNREAS, BMLOC, BSLOC는 커버 가능. label이 비어 있는 table block은 제외됨.
UNKNOWN 1개 0개 미커버 자동 분류 또는 실행 가능한 step 생성 전 manual review 필요.

8. 다음 개선 방향

현재 DM 페이지에서는 단순 radio/date 기반 query와 availability 조건은 자동화 가능성이 높다. 다음 개선 우선순위는 다른 페이지/선행 데이터 참조, assign/calculate trigger, appendable table 내부 field 매칭을 보강하는 것이다. 특히 유방암 병력/수술력 appendable table의 날짜 field와 item id 연결이 확보되면 현재 불가능으로 분류된 D_BM, D_BS 계열 query의 커버리지가 높아진다.

appendable table은 일반 row와 달리 row header, column header, 반복 row index, add/remove row 동작이 함께 얽혀 있다. 따라서 별도 개발 축으로 분리해 row 생성, 반복 row 선택, column별 control 탐색, item id 매칭, candidate step 생성을 단계적으로 구현해야 한다.
